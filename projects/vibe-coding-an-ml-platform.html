Vibe-coding an ML Platform

Published: July 2, 2025

If you've been following my blog, you know I've been trying to gauge the effectiveness of SWE-agents for a variety of tasks across the stack. Naturally I wanted to see how well they do in building out infrastructure as code. To test this out, I set out to build a minimalistic ML platform. I think of this as a box of scalable software infrastructure that I can take from one Virtual Private Cloud to another and get a repeatable scalable toolkit.

Before I started, here's what I consider to be the minimalist basics of an ML platform. There's a lot more on the data side but I'm going to focus on minimalism for now.

  • Experimentation Tools
  • Model Artifact Management
  • Model Serving
  • Model Monitoring / Evals
  • Model Versioning

To be honest, not much has changed in the GenAI world compared to traditional ML besides the UX of the tooling. The broader componentary remains the same.

Essentially I wanted to combine several open source projects to build a minimalistic ML platform that satisfies the above requirements. Additionally I wanted to roll out deployments through a gitops workflow. For example, let's say I needed elastic distributed computing to run hyperparameter tuning experiments for some ML problem. I wanted the ability to make a commit to a repo to introduce a ray cluster as a service to my ML Platform.

To my surprise, within a few hours I had a gitops based kubernetes cluster managed by ArgoCD that deploys a JupyterHub instance with a Ray Cluster where I can immediately start experimenting at scale for any machine learning use-case. Check out the architecture doc here

This is a work-in-progress and I'll continue to build out more features as I go. I also hope to deploy some sample ML / GenAI apps to really put this seminal paper to the test: Hidden Technical Debt in Machine Learning Systems. The question is - In 2025 with modern open-source abstractions and SWE-Agent driven productivity, can team's (and individuals) be so nimble that technical debt in these systems doesn't really matter anymore. I imagine a world with hyper-personalized deployments, tailored to each use-case built on abstractions that these LLMs understand well.
