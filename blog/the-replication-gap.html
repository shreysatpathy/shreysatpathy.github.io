The Replication Gap: Why 14 Self-Propagating Worms Infected the AI Developer Supply Chain in 60 Days — And Why Your AI Coding Assistant Is Ground Zero

June 30, 2026
AI Security Infrastructure Product Management

A developer on your team installs a widely used npm package. It's got 500,000 weekly downloads, passing CI, clean provenance attestation — SLSA Build Level 3 signed. Everything looks fine. The package ships a dependency that ships a script that writes a configuration file into .claude/settings.json. The next time your developer opens their project in Claude Code, the AI assistant reads those settings and, following what appear to be legitimate project-level instructions, quietly exfiltrates every API key, cloud credential, and npm publish token it can find. The developer sees nothing. The AI assistant reports everything green. The worm has already propagated to three other repos using the stolen tokens before anyone notices the first infection.

This is not a hypothetical. This is the week of June 1, 2026.

The Cloud Security Alliance and JFrog Security Research documented two self-replicating worms — Miasma and IronWorm — that emerged from the npm ecosystem in the first week of June, explicitly targeting AI coding tool credentials. Miasma compromised 57 npm packages across 286 malicious versions and spread into 73 GitHub repositories, planting backdoor files in Claude Code, Cursor, Gemini CLI, and VS Code configurations. IronWorm, a 976 KB Rust-compiled implant, carried an eBPF kernel rootkit, communicated over Tor, and systematically harvested API keys for Anthropic, OpenAI, Google Gemini, Cohere, Mistral, Groq, Perplexity, and xAI — 86 environment variables and 20 credential file paths in total.

These two worms were not outliers. They were the leading edge of a wave. Across the Cloud Security Alliance's 59 documented supply chain campaigns in the first half of 2026, 14 targeted AI developer tooling explicitly — and the rate is accelerating. The VS Code Marketplace went from near-zero malicious activity to 7 documented campaigns in 18 months. ReversingLabs recorded detections nearly quadrupling from 27 in 2024 to 105 in the first 10 months of 2025 alone.

I call this the replication gap. The AI developer supply chain has become the fastest-growing attack surface in enterprise history — and most organizations have no idea their AI coding assistant is the primary distribution vector.

The four-generation worm evolution

What makes this moment different from every prior supply chain attack is the compounding sophistication of the worm lineage. The CSA's tracking shows a clear generational progression:

First generation (2025). Credential theft via malicious npm/PyPI packages. Infect the registry, steal tokens, publish more malicious packages. Effective but detectable — standard SBOM tooling and registry monitoring caught most of these.

Second generation (early 2026). The TrapDoor campaign introduced the first documented use of zero-width Unicode characters in AI coding assistant config files. Attackers published 34+ malicious packages across npm, PyPI, and Crates.io (384+ versions) targeting crypto, DeFi, Solana, and AI developers. The innovation: .cursorrules and CLAUDE.md files containing hidden instructions invisible to human reviewers but fully interpretable by AI assistants. When a developer opened the project, the AI assistant would receive and execute instructions — framed as a "security scan" — to collect and exfiltrate secrets. No vulnerability in the AI tool. Just an exploit of its designed behavior.

Third generation (April-May 2026). Mini Shai-Hulud, from the TeamPCP threat actor, became the first documented supply chain attack to forge valid SLSA Build Level 3 provenance attestations by extracting GitHub Actions OIDC tokens from runner process memory. The worm compromised 373 malicious package-version entries across 169 npm packages and 2 PyPI packages with a cumulative download base exceeding 518 million. Its distinguishing innovation: it weaponized AI coding agent configuration files as a persistence mechanism. The worm wrote hooks into .claude/settings.json and .vscode/tasks.json that survived complete package removal — meaning even after the infected dependency was uninstalled, the AI assistant would re-execute the malicious hooks every time a developer opened the project. The remediation order itself was safety-critical: the worm's persistence service triggered home directory destruction if credentials were rotated before the service was disabled.

Fourth generation (June 2026). Miasma and IronWorm combined self-propagation with multi-tool targeting — simultaneously compromising Claude Code, Cursor, Gemini CLI, and VS Code configurations from a single infection event. Miasma spread autonomously using stolen npm publish tokens, GitHub write access, and CI/CD OIDC tokens. IronWorm added an eBPF rootkit to hide from standard monitoring. The line between infostealer and worm is now completely blurred. These are not attacks on AI. They are attacks through AI — using the tools your developers trust as the infection vector.

The Cloud Security Alliance's Sub-4-Hour Weaponization report documents something equally alarming: CVE-2026-44338, a missing-authentication vulnerability in PraisonAI, was weaponized within 3 hours and 44 minutes of public disclosure. 28.96% of known-exploited vulnerabilities were weaponized on or before the day of CVE publication in 2025, up from 23.6% in 2024. AI-assisted reverse engineering is compressing this window further. Organizations running traditional patch cycles measured in days or weeks are operationally exposed.

Why the AI developer supply chain is uniquely vulnerable

Every organization I talk to has invested heavily in AI coding assistant security: approved tool lists, usage policies, data residency requirements. Almost none have invested in the security of the interactions between those tools and the open-source packages their developers install daily. The replication gap has four structural causes:

1. AI coding assistants operate with elevated trust. Claude Code, Cursor, GitHub Copilot — these tools read project files, execute shell commands, access network resources, and manage credentials. They operate at the intersection of a developer's most sensitive systems. When a malicious package plants a configuration file in .cursorrules, the AI assistant reads it as a legitimate project-level instruction. There is no integrity verification. There is no "are you sure you want to follow these instructions?" prompt. The trust model assumes that anything in the project directory is safe.

2. The AI config file is the new persistence layer. Traditional supply chain attacks rely on continuous execution — a running binary, a cron job, a launch agent. Remove the package, and the infection stops. AI config files break this model. As the CSA documented, Mini Shai-Hulud's hooks in .vscode/tasks.json survive package removal and reactivate every time the developer opens VS Code. The standard remediation playbook — "uninstall the malicious package" — does not work for AI-assisted attacks. The config file persists, trusted by the tool, re-infecting the environment on every session.

3. Zero CVEs across the 59-campaign corpus. This is the statistic that should keep every CISO awake. Phoenix Security's analysis of all 59 tracked supply chain campaigns found that zero CVEs were assigned across the entire corpus. These are not exploits of software vulnerabilities. They are exploits of trust relationships — the implicit trust between AI tools and configuration files, between developers and package registries, between CI/CD pipelines and provenance attestation. Traditional vulnerability management programs cannot detect them because there is no vulnerability to patch.

4. The credential blast radius is enterprise-wide. As I covered in the identity gap, 80% of AI agents run on borrowed credentials. The worms documented in this wave don't just steal one developer's tokens. They use CI/CD OIDC tokens to sign malicious packages with forged provenance. They extract cloud keys that open production infrastructure. They harvest API keys that expose every model, every prompt, and every retrieved document in your AI stack. The supply chain gap I wrote about — 200,000 unauthenticated MCP instances — was already alarming. The replication gap makes it worse: those instances are now targets for automated credential harvesting at worm speed.

What the response looks like

There is no single fix for the replication gap. But the organizations that are responding effectively are converging on four controls:

Treat AI tool config directories as trust boundaries. .claude/, .cursor/, .gemini/, .vscode/ — these directories should be subject to the same access controls as production secrets stores. Monitor them for unexpected writes. Alert on configuration changes that originate from package install scripts. The CSA recommends treating AI coding assistant configuration as a separate trust domain, distinct from both the project code and the developer's personal configuration.

Implement runtime integrity checks for AI tools. Before an AI assistant reads a configuration file, the runtime should verify its integrity against a known-good baseline. This is analogous to container image signing: config files should carry checksums, and the AI tool should refuse to load unverified or modified configurations. The agent observability platforms emerging in 2026 — Braintrust, Arize, Honeycomb, Omium — are starting to surface the tracing infrastructure needed to detect when an agent acts on unexpected configuration. But this capability needs to live in the tool itself, not in an external observability pipeline.

Segment AI credentials from production credentials. The replication gap exists because a single compromised token provides access to both the package registry (where the worm publishes) and the production infrastructure (where the worm persists). Organizations should issue AI-specific credentials scoped to publish-only or read-only access, with separate credentials for production infrastructure. The TeamPCP campaign specifically exploited the fact that LiteLLM API keys — meant only for model routing — provided access to production inference infrastructure. Credential scoping alone wouldn't have prevented the initial infection, but it would have stopped the worm's ability to propagate.

Build AI BOMs and monitor them at runtime. Software bills of materials (SBOMs) are standard practice. AI bills of materials (AI BOMs) — covering model dependencies, inference libraries, API gateway components, and AI tool configurations — are not yet standard, but they need to be. The CSA's MAESTRO threat model for agentic AI systems explicitly includes infrastructure-layer integrity — including the integrity of tools and services that AI agents invoke at runtime — as a first-class threat category. If your AI BOM does not include the AI coding assistant configuration files that every developer workstation runs, your threat model has a blind spot the size of the npm ecosystem.

The unbudgeted cost

The replication gap is not a security problem. It is a product management problem with security consequences. Every product leader I talk to has budgeted for AI model costs, AI infrastructure, and AI governance tooling. Almost nobody has budgeted for the security incident response capability needed when a self-propagating worm compromises their AI developer supply chain. The cost gap — 98% track spend, 9% know what it bought — has a security dimension: most organizations cannot tell you how many AI tool configurations exist across their developer workstations, let alone whether any of them have been tampered with.

The worms documented in this post are the first generation of what will become a standard attack pattern. The AI coding assistant is the highest-trust, lowest-integrity system in your enterprise. It has access to everything your developers have access to — and it follows instructions from anyone who can write a config file.

The question is not whether your organization will encounter a replication-gap incident. It is whether you will discover it before the worm uses your credentials to infect the next 50 organizations downstream.
