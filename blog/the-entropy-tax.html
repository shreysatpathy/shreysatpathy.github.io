The Entropy Tax: Why Non-Determinism Is the Most Expensive Hidden Cost in Agentic AI

June 17, 2026
AI Reliability Infrastructure Product Management

I watched a team run the same agent on the same input three times in a row last week. First run: 12 seconds, 4 model calls, correct answer. Second run: 47 seconds, 11 model calls, hallucinated a tool. Third run: timed out after 90 seconds and 23 model calls. Same code. Same prompt. Same input. Three completely different outcomes.

Nobody on the team was surprised. That's the terrifying part.

We've normalized non-determinism in agentic systems. We've accepted that running the same agent twice produces different results, different costs, and different reliability characteristics — and we've built our deployment strategies around this assumption rather than treating it as the crisis it actually is. The [OutSystems 2026 State of AI Development report](https://www.outsystems.com/news/enterprise-ai-agent-report-2026) found that 96% of organizations are already using AI agents, but 94% report concern about sprawl and complexity. Non-determinism is the engine driving both.

I call this the entropy tax: the compounding cost of unpredictable behavior in every agent interaction. It shows up in ways most teams aren't measuring.

Three faces of the entropy tax

1. Cost unpredictability. The same workflow can cost 5x more depending on which reasoning path the model takes. A straightforward data extraction might resolve in 3 calls ($0.02). The same extraction, on a day when the model decides to second-guess its tool selection, can spiral into 15 calls with retries ($0.30). At scale, this variance makes budgeting impossible. Finance asks "what's our per-transaction cost?" and the honest answer is "it depends on how the model feels."

2. Debugging chaos. Traditional debugging assumes reproducibility. Show me the logs from the failed run, and I can trace the execution. Agentic systems don't work that way. The failure path from run #2 doesn't exist in run #3. [As I covered in the observability blindspot](the-observability-blindspot.html), most APM tools collapse an agent's reasoning into a single black-box span. When every failure is a unique snowflake, pattern detection becomes statistical rather than forensic. You can't fix what you can't reproduce.

3. Reliability math that doesn't add up. If a single agent step succeeds 85% of the time (generous by most production measurements), a 10-step workflow succeeds roughly 20% of the time on any given run — but that's the average. The [70-95% failure rate I documented in the reliability gap](the-reliability-gap.html) masks a deeper problem: the variance from run to run is as dangerous as the mean. Some runs succeed in 3 steps. Others degrade into 20-step death spirals. The same agent, same task, same hour — completely different behavior.

The [Presenc AI Q1 2026 survey](https://presenc.ai/research/enterprise-ai-adoption-statistics-2026) found that 67% of enterprises cannot explain why their agent produced a specific output. That's not a governance gap. That's an entropy problem. You can have perfect policies and still fail to govern systems whose behavior you can't predict on a per-run basis.

What the reliable teams do differently

The organizations running multi-agent systems in production at scale — the ones who've crossed the chasm from pilot to reliable operation — have adopted a different mental model. They treat non-determinism as a first-class engineering constraint, not a philosophical property of LLMs.

Budgets, not guesses. Every workflow has a hard token budget, a model call budget, and a time budget. If the agent exceeds any of these thresholds mid-execution, it fails fast rather than wandering. The cost cap becomes a forcing function for prompt design. When every path must fit within 5 calls, you learn to stop asking the model to "figure it out" and start telling it exactly what to do.

Deterministic scaffolding. The top-quartile teams extract as much logic as possible from the LLM and encode it in code. Routing, validation, error handling — anything that can be expressed as a conditional or a schema check gets pulled out of the prompt. The model only does what models do well: reasoning over ambiguous inputs. Everything else is hard-coded. This sounds obvious. It is not what most teams do.

Run-to-run comparison as a core metric. These teams track how often the same input produces a different output, and they treat divergence as a bug. If a customer asks "what's my balance?" and the agent answers correctly 8 out of 10 times, they don't celebrate the 80% — they investigate the 20%. The entropy tax is measured in per-workflow variance reports, not averages.

Observability at the decision level. Standard monitoring tells you an agent completed its task. The [agentic observability tools emerging in 2026](https://www.braintrust.dev/articles/agent-observability-complete-guide-2026) — Fiddler, Honeycomb, Braintrust, Arize — are starting to surface reasoning traces that show which path the agent took, not just that it finished. This is the difference between a speedometer and a flight recorder. Teams that can replay the exact decision path for any run can fix entropy at the source.

The budget question nobody asks

Every PM I know has a line item for model inference costs. Almost nobody has a line item for the cost of debugging non-deterministic failures, or the cost of building deterministic scaffolding, or the cost of variance in per-transaction costs that makes unit economics impossible to calculate.

The [cost gap](the-cost-gap.html) — where 98% of organizations track AI spend but only 9% can tell you what it bought — has a non-determinism dimension. When the same workflow costs $0.02 on one run and $0.30 on another, which number goes in the budget? Most teams use the average. The average is a lie.

The entropy tax is real. It's the unbudgeted cost of building on a foundation that behaves differently every time you touch it. The teams that acknowledge it, measure it, and engineer around it are the ones shipping agentic systems that actually work. Everyone else is running the same agent three times and hoping for different results.
