The Identity Gap: Why 80% of AI Agents Are Running on Borrowed Credentials

June 17, 2026
AI Security Governance Product Management

Here's a question I've started asking every AI engineering team I meet: "When your agent calls an API, whose credentials is it using?"

The answers are a masterclass in creative security debt. "It uses the service account." "It's embedded in the config file." "We haven't gotten to that yet." One team told me, honestly, "I don't actually know."

They're not alone. Akeyless's 2026 AI Agent Identity Security Survey found that fewer than half of organizations know where all their AI agent credentials are stored, and over 80% admit their agents regularly bypass IAM controls. Gravitee's State of AI Agent Security 2026 puts the number of organizations that treat AI agents as independent, identity-bearing entities at just 21.9%.

Nearly 80% of AI agents in production are running on borrowed credentials. Shared API keys. Standing service accounts. Tokens copy-pasted into environment files. And every single one of them has access to more than it needs.

The identity debt spiral

Identity debt in AI agents compounds faster than any technical debt I've seen. Here's why: in traditional software, a microservice inherits the identity of the pod or the VM it runs on. It's not great, but the blast radius is contained — a service can only do what its API endpoints allow. An AI agent is different. An agent doesn't have a fixed set of endpoints. It decides what to call based on reasoning. If an agent inherits a service account with broad permissions, it has de facto access to everything that account can reach. And the agent might discover capabilities the original developer never anticipated.

The Cloud Security Alliance's April 2026 study found that 53% of organizations have had AI agents exceed their intended permissions. Only 8% said agents never exceed their scope. When scope violations occur — and they will — only 11% of organizations automatically block the action. The rest log it and hope someone reviews the logs.

This is the pattern that keeps SOC analysts up at night: an agent scrapes a Jira ticket for context, finds an API key in a comment, and uses it to pull customer PII from a data warehouse. Was that intended behavior? In a post-mortem, maybe. At runtime, there's no way to tell.

I wrote about the enforcement gap last week — the 68-point chasm between policy confidence and runtime controls. The identity gap is the enforcement gap's root cause. You cannot enforce policy against an agent you cannot identify.

Why nobody's solving this

Three structural reasons explain why identity for AI agents is stuck in 2020 while the agents themselves are racing toward 2027.

1. Agent identity doesn't fit the IAM model. Traditional IAM was built for humans and service accounts. Humans get a unique ID, MFA, and revocable sessions. Service accounts get a static secret and a scope. An AI agent is neither. It's ephemeral, it's stateful, it calls tools dynamically, and its behavior changes based on the prompt. IAM systems weren't designed for actors whose scope of action is determined by a model call, not a config file. Every identity team I've talked to is trying to retrofit agent identity into a framework built for cron jobs. It doesn't fit.

2. Speed of deployment trumps speed of security. The Gartner forecast — 40% of enterprise apps embedding task-specific agents by end of 2026 — means engineering teams are shipping agents as fast as they can build them. Identity is an afterthought because it's not on the critical path to demo. The agent works with a shared key. Identity is "a future sprint item." That sprint never comes because the next agent is already in progress.

3. Nobody wants to own the problem. IAM teams own identity. AI/ML teams own agents. Platform teams own the runtime. Security teams own policy. The agent sits at the intersection of all four, and none of them want to be the one that slows down AI adoption by insisting on identity controls. The result is a collective action problem: everyone agrees agents need identity. Nobody is willing to be the bottleneck that enforces it.

What agent identity looks like in practice

The 21.9% of organizations that do treat agents as identity-bearing entities share a common architecture. It's not complicated, but it's deliberate:

Every agent gets a unique, scoped identity at creation time. Before the agent makes its first tool call, it's registered in the identity system with a purpose, a permission boundary, and a max TTL. The identity is ephemeral — if the agent doesn't complete its task within the window, the identity expires. No standing credentials, no shared keys, no "we'll clean this up later."

Every tool call is checked against the identity's scope. When the agent calls an API, the authorization layer evaluates the call against the agent's registered scope — not the scope of some underlying service account. The service account still exists underneath, but it's never exposed to the agent. The authorization layer translates the agent's intent into scoped, auditable actions. If the agent tries to call something outside its scope, the call is denied before it reaches the API. This is the pre-execution control I described in the enforcement gap — applied at the identity level.

Every action is attributable. Because every agent has a unique identity, every tool call, every API response, every data access is traceable to a specific agent instance. Not a "service account" or a "pod" — a specific agent, running a specific task, initiated by a specific user or trigger. This is the audit trail regulators are starting to ask for. ISO 42001 adoption is accelerating — 31% of enterprises now hold certification — and auditability is a core requirement.

The Akeyless survey found that organizations implementing these patterns reduced credential-related incidents by over 60%. The cost is real — identity infrastructure for agents requires mapping permissions, wiring authorization layers, and changing deployment workflows. The alternative is an agent fleet where every member has the keys to the kingdom and nobody knows which door it just opened.

The regulatory clock is ticking

The EU AI Act is in effect. The SEC is watching. Gartner's May 2026 warning that 40% of enterprises may demote or decommission autonomous agents by 2027 cited inadequate risk controls as a root cause — and identity is the foundational risk control. You cannot audit what you cannot identify. You cannot contain what you cannot distinguish from legitimate traffic. You cannot trust an agent fleet where 80% of members are wearing someone else's badge.

The most striking number in the Gravitee survey isn't the 88% who've had security incidents — it's the 21.9% who treat agents as identifiable entities. That 78-point gap is the size of the identity debt the industry has already accumulated. And unlike code debt, identity debt can't be refactored after a breach.

In The Great Cheapening, I wrote about how 280x cheaper inference made agentic AI economically viable. The cheapening created the boom. Identity is the infrastructure that determines whether the boom leaves a trail or a crater. Every agent running on borrowed credentials today is an incident waiting for a trigger. The question isn't whether your agents will be audited — it's whether you'll have the identity infrastructure to survive the audit.
