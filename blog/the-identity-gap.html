The Identity Gap: Why 80% of AI Agents Are Running on Borrowed Credentials

June 17, 2026
AI Security Governance Product Management

I've started asking every AI engineering team I meet a simple question: "When your agent calls an API, whose credentials is it using?"

The answers are a masterclass in creative security debt. "It uses the service account." "It's embedded in the config file." "We haven't gotten to that yet." One team told me, straight-faced, "I don't actually know."

Turns out they're not alone. Recent surveys show over 80% of organizations admit their agents regularly bypass IAM controls. Only about 22% treat AI agents as independent, identity-bearing entities. That means nearly 80% of AI agents in production are running on borrowed credentials. Shared API keys. Standing service accounts. Tokens copy-pasted into environment files. And every single one has access to more than it needs.

Why this compounds faster than technical debt

In traditional software, a microservice inherits the identity of the pod it runs on. Not ideal, but the blast radius is contained — a service can only do what its endpoints allow. An AI agent is different. It doesn't have fixed endpoints. It decides what to call based on reasoning. If it inherits a service account with broad permissions, it has access to everything that account can reach. And the agent might discover capabilities the original developer never anticipated.

More than half of organizations have had AI agents exceed their intended permissions. When scope violations happen, only 11% automatically block the action. The rest log it and hope someone reviews the logs. This is the pattern that keeps security analysts up at night: an agent scrapes a Jira ticket, finds an API key in a comment, and uses it to pull customer PII from a data warehouse. Was that intended? At runtime, there's no way to tell.

Three reasons nobody's solving this

First, agent identity doesn't fit the IAM model. Traditional IAM was built for humans and service accounts. An AI agent is neither — it's ephemeral, stateful, calls tools dynamically, and its behavior changes based on the prompt. Every identity team I've talked to is trying to retrofit agent identity into a framework built for cron jobs. It doesn't fit.

Second, speed of deployment trumps speed of security. Teams are shipping agents as fast as they can build them. Identity is "a future sprint item." That sprint never comes because the next agent is already in progress.

Third, nobody wants to own the problem. IAM teams own identity. AI teams own agents. Platform teams own the runtime. Security teams own policy. The agent sits at the intersection of all four, and none of them want to be the one that slows down AI adoption. Everyone agrees agents need identity. Nobody is willing to be the bottleneck.

What it looks like when it works

The organizations that do this right share a simple architecture. Every agent gets a unique, scoped identity at creation time — before it makes its first tool call. The identity is ephemeral: if the agent doesn't finish within a window, the identity expires. No standing credentials. No shared keys. Every tool call is checked against the agent's scope, not some underlying service account. And every action is attributable to a specific agent instance running a specific task.

Organizations that implement these patterns reduce credential-related incidents by over 60%. The cost is real — it requires mapping permissions, wiring authorization layers, changing deployment workflows. The alternative is an agent fleet where every member has the keys to the kingdom and nobody knows which door just opened.

The EU AI Act is in effect. Regulators are starting to ask questions. You cannot audit what you cannot identify. You cannot contain what you cannot distinguish from legitimate traffic. The question isn't whether your agents will be audited — it's whether you'll have the identity infrastructure to survive that audit.
