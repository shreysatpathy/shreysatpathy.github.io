The Reliability Gap: Why 70-95% of AI Agents Fail in Production — And Why the Model Isn't the Problem

June 26, 2026
AI Infrastructure Reliability Product Management

Your agent completed the demo perfectly. Then it created 847 duplicate customer records in production before anyone noticed the retry loop. It returned HTTP 200 on every call. Token usage looked normal. And yet somewhere downstream, a payment was never processed, a CRM record was never updated, and a customer is about to file a chargeback.

I call this the reliability gap. It is the difference between what an agent does in a controlled demo and what it does in a live workflow. And it is the most expensive blind spot in enterprise AI right now.

Fiddler AI's 2026 production analysis found that AI agents fail between 70% and 95% of the time in real-world settings, depending on task complexity and how success is measured. On the WebArena benchmark, the best GPT-4-based agent achieved an end-to-end task success rate of only 14.41%, compared to human performance of 78.24%. Carnegie Mellon researchers found that AI agents fail at common office tasks roughly 70% of the time. An MIT report found that 95% of generative AI pilots fail to deliver measurable impact on the P&L.

The common thread across every study: the model is not the bottleneck. The reliability infrastructure is.

The math that kills multi-step agents

Traditional software is deterministic. The same input produces the same output every time. AI agents are probabilistic — and they chain multiple probabilistic steps together. This creates a compound failure problem that most teams do not account for until their agent is already in production.

Temporal's engineering team documented the math precisely. If each individual agent step is 85% reliable, a 10-step workflow succeeds end-to-end only about 20% of the time. Even at 95% step reliability — which most teams would consider excellent — a 15-step workflow succeeds just 46% of the time. Most enterprise workflows have more than 15 steps.

A 2025 survey of AI agent practitioners cited in the AgentMarketCap function calling reliability report confirmed the pattern: cascading failure dominates multi-step workflows. The silent function call — where the agent returns a convincing summary of what it did without actually doing it — is the most common, least-tracked class of bug in production AI agent systems. Unlike crashes that produce stack traces, silent failures produce plausible-looking outputs while quietly skipping the actual work. An agent that sounds right but does nothing is worse than an agent that crashes, because the crash gets noticed, triaged, and fixed.

The four failure modes nobody builds for

1. Ghost actions. The agent tells the user it completed a task — booked the flight, processed the refund, filed the report — but never actually called the underlying API. The response sounds perfect. The action never happened. LLMs are trained to produce helpful, complete-sounding outputs. When tool execution fails silently or the model skips a step, it generates a confident confirmation anyway. Traditional output-only evaluation scores these responses highly because they are fluent and relevant. Ghost actions accounted for roughly 34% of production agent failures in the Fiddler AI analysis.

2. Compound cascades. A single misstep early in a multi-step workflow propagates through every downstream step. A tool call that returns a subtly wrong value — a timestamp instead of ISO 8601, a customer ID that is technically valid but wrong — causes every subsequent step to operate on corrupted data. The agent detects nothing wrong because each individual call succeeded. The 85% per-step reliability math means that most multi-step agents will hit a cascade failure within their first week of production operation.

3. Retry loops and cost runaway. An agent that cannot complete a step will retry. Then retry again. Then retry with a different approach. Then retry the original approach. Without upper bounds on retry counts and token consumption, agents burn through inference budgets in minutes. Apptitude's 2026 production analysis documented cases of agents generating $500+ in token costs from a single runaway retry loop before the team noticed the cloud bill. The agent does not know it is looping. It believes it is making progress.

4. Context window poisoning. Long-running agents accumulate context. The more steps, the more tool outputs, error messages, and intermediate results fill the context window. At some point, the agent's relevant context is diluted by its own operational history. StackPulsar's reliability guide recommends alerting when context utilization exceeds 80%, but most teams do not monitor context utilization at all. The agent does not fail suddenly. It degrades gradually, producing increasingly erratic outputs as relevant signal is buried under noise.

Why the model is not the answer

The reflex reaction to reliability problems is to switch to a better model. GPT-4o instead of GPT-4o-mini. Claude 4 instead of Claude 3.5. Better models do improve step-level reliability — but the improvement is incremental, not transformational. A model that improves per-step accuracy from 85% to 90% changes a 10-step workflow's end-to-end success rate from 20% to 35%. That is better. It is not production-grade.

The 2026 International AI Safety Report, authored by over 100 experts, identifies persistent unreliability as a core challenge for the foundation models underpinning agentic systems. This is not a solvable-next-quarter problem. It is an architectural challenge you design around.

The organizations that close the reliability gap do not wait for better models. They build the infrastructure layer that makes probabilistic systems behave deterministically. The Agent Harness Engineering survey published at TMLR 2026 found that modifying only the tool harness — with no model changes — produced reliability gains of up to 10 percentage points on coding benchmarks across 15 different models. Infrastructure, not the model, is the lever.

How to close the reliability gap

1. Treat reliability as a numbers game, not a quality problem. Map every workflow. Count the steps. Calculate the compound reliability. If your per-step accuracy is 90% and your workflow has 15 steps, your end-to-end success rate is 21%. That is your baseline. Do not deploy until you know that number. Every team that has closed the reliability gap started by admitting the math was against them.

2. Build verification layers, not just output evaluation. Output evaluation scores the response. Verification confirms the action. These are different things. A verification layer checks that the tool call actually executed, that the downstream system state changed as expected, and that the result satisfies the task's success criteria — not just the model's fluency criteria. The teams that achieve production-grade reliability separate evaluation from verification and treat them as independent systems.

3. Implement trace-level observability. You cannot fix what you cannot see. Every agent action — every LLM call, every tool invocation, every retry, every decision branch — needs a trace with span-level detail. Grafana's recommended AI agent dashboard includes five panels: task success/failure rate, token burn rate by agent, tool call latency histogram, step count per task, and context window utilization. Most teams monitor none of these. The teams that close the reliability gap monitor all five before they deploy.

4. Start with copilot, graduate to autonomous. The organizations that succeed with AI agents do not deploy fully autonomous agents on day one. They start with a copilot design — the agent proposes, the human disposes. High-stakes decisions require confirmation. Tool calls above a cost threshold require approval. This is not cowardice. It is training data. Every human-approved or human-rejected action generates a labeled data point that improves the agent's reliability over time. Carnegie Mellon's research found that agents with human-in-the-loop training improved their success rates by 27 percentage points over four weeks of supervised operation.

5. Cap your retries and your context. Every agent needs hard limits. Maximum retry count per step. Maximum token spend per task. Maximum context utilization before forced summarization. These are not engineering luxuries. They are safety guarantees. The $500 retry loop happened because nobody set a budget. The 847 duplicate records happened because nobody set a retry cap. The gradually degrading agent happened because nobody was watching the context window. Hard limits do not prevent success. They prevent uncontrolled failure.

The reliability gap is the purest expression of the gap between demo and production. Every organization in my "Gap" series has the same root cause: we deployed AI agents before we built the infrastructure to make them reliable. The model works. The demo works. The production system does not — because production is where probabilistic systems meet deterministic expectations, and the harness between them is not ready.

Closing the reliability gap does not require better AI. It requires better engineering. Map your workflows. Count your steps. Build your verification layers. And deploy with your eyes open about the math. The model will improve. Your reliability infrastructure has to improve faster.
