The Injection Gap: Why 85% of AI Coding Agents Execute Stranger Instructions

June 18, 2026
AI Security Governance Product Management

You deploy an AI coding agent. It reads a Sentry error log, finds what looks like a stack trace, and follows instructions embedded inside it. The instructions tell it to modify your authentication module. The agent executes them. You now have a backdoor in production.

This isn't a hypothetical. It's Agentjacking — a new class of supply chain attack that exploits the one thing every AI agent does without question: trust the data it reads. AI Weekly broke the story on June 16, reporting that crafted fake Sentry error messages achieve an 85% exploitation rate against Claude Code, Cursor, and Codex. 2,388 organizations have been identified as exposed. A self-replicating worm hit 73 of Microsoft's own GitHub repositories through AI coding tools earlier this month.

This is not prompt injection as we knew it in 2024. Classic prompt injection was about making a chatbot say something embarrassing. Agentjacking is about making a coding agent write malicious code into your codebase and push it through your CI/CD pipeline. The agent has read and write access. The attacker never touches your network. They just need to leave a comment on a public issue, a log entry in a dependency, or a markdown file in a repo your agent will crawl.

The graph here is sobering: with every new capability we give agents — file write, code execution, PR creation, production deploy — we widen the blast radius of the oldest unsolved problem in AI security.

Why agents obey strangers

The root cause is architectural. AI agents today are built with an implicit trust assumption: data from the environment is distinct from instructions from the user. But at the model level, there is no distinction. A sentence in a Sentry log and a sentence in a system prompt are both just tokens. The model treats them identically because the architecture has no mechanism to say "this text is content, that text is a command."

This is the same problem that makes SQL injection possible, except SQL injection was solved in the 2000s by parameterized queries. We separated data from code. We have no equivalent for LLM agents. Every token an agent consumes is treated as potentially instructional, because at the inference level, tokens are tokens. The boundary between "data" and "prompt" doesn't exist in the transformer architecture.

Researchers have known this since at least 2023. The OWASP Top 10 for LLM Applications has had prompt injection at #1 since its first release. But the industry treated it as an LLM problem — something for prompt engineers to worry about — rather than a software supply chain problem. Agentjacking is the consequence of that neglect. When your agent is the one writing code, prompt injection isn't a jailbreak. It's an arbitrary code execution vulnerability with a 6-second time-to-exploit.

I wrote last week about the identity gap — the 78-point chasm between agents that have unique identities and those that don't. The injection gap is different. Even if every agent had a perfect, scoped identity, the agent would still execute injected instructions because the architecture trusts the data channel. Identity tells you which agent did what. It doesn't stop the agent from doing the wrong thing.

Why the runtime is the only place to fix this

This brings me to the most important strategic development of June 2026, which I believe will be remembered as the month the industry stopped selling models and started selling agent runtimes. At Microsoft Build, the message was Agent 365 and Azure AI Foundry as orchestration hubs. At Nvidia's GTC Taipei, it was an open-source Agent Toolkit with a secure runtime. At ServiceNow Knowledge, it was governed autonomous agents with auditability. Google's Managed Agents in the Gemini API spin up agents inside isolated sandboxes.

The common thread: all of them are selling the environment an agent lives in, not the intelligence inside it. The model is commodity. The runtime is the product. And the runtime is also the only layer where injection attacks can be stopped — because the runtime sits between the agent and every tool call, every data read, every file write.

A secure agent runtime does three things that the current generation of agents skips:

1. Content provenance verification. Before an agent acts on data it reads, the runtime validates where the data came from. A Sentry log from a known endpoint with a valid signature is one thing. A comment on a public GitHub issue posted 3 minutes ago by an account with no history is something else. The runtime attaches a provenance tag to every piece of data the agent consumes, and the agent's tool-call permissions are gated on the trust level of the provenance. Google's expansion of SynthID watermarking to over 100 billion assets, with OpenAI and Eleven Labs adopting the standard, suggests the industry recognizes this direction. Content provenance for agent inputs is the natural extension.

2. Instruction-content separation at the tool-call boundary. When an agent calls a tool, the runtime inspects the arguments for patterns consistent with injected instructions — embedded system prompts, delimiter manipulation, known jailbreak templates. This is the parameterized query equivalent for agents. It's not perfect, but it raises the cost of exploitation from "leave a comment" to "bypass a detection layer." The enforcement gap I described — policy-as-code at the runtime level — applies here directly. The policy just needs to include injection detection as a first-class control.

3. Constrained action surfaces. Instead of giving an agent free rein to write files, the runtime defines a blast-radius budget per agent instance. An agent analyzing bug reports doesn't need write access to the authentication module. The runtime enforces that constraint even when the agent's reasoning decides it does need that access. This is the identity gap's enforcement layer, but applied to action scope rather than credential scope.

What the numbers tell us

The 85% exploitation rate on Agentjacking is not an outlier. It's the expected result when you give agents broad tool access without runtime security. The self-replicating worm in Microsoft's repos is the shape of things to come: an injected payload that convinces an agent to write code that, once deployed, injects the same payload into the next agent that reads its logs. A self-propagating agent worm is not science fiction. It's June 2026.

The AI coding agent market is the fastest-growing segment in enterprise AI. Microsoft's own internal data showed 84-95% monthly usage of Claude Code licenses in its Experiences and Devices division before canceling them. Uber's engineering teams burned through the entire 2026 AI coding budget in four months. At Meta, employees track "Claudeonomics" leaderboards to compete on token consumption. The adoption curve is vertical. And the security model is still running at 2023 assumptions.

The EU AI Act's main rules take effect on August 2, 2026. Article 15 requires "accuracy, robustness, and cybersecurity" for high-risk AI systems. A system with an 85% injection exploitation rate does not meet that standard. The SEC is watching. Regulators are going to start asking: what runtime controls do you have in place to prevent your AI agents from executing arbitrary instructions from untrusted sources? If the answer is "we trust the model not to follow bad instructions," the conversation will be short.

The path forward

I've traced twelve structural barriers to production AI in this series. They all converge on the same conclusion: the agent runtime is the product, and security is the runtime's killer feature. Models will keep getting cheaper — the 280x cost collapse was just the beginning. Tooling will keep getting better. But until we solve the injection gap — until we build runtimes that can tell the difference between a legitimate instruction and a stranger's payload — every agent in production is a supply chain attack waiting for a trigger.

The 85% number should terrify you. It should also focus you. The runtime is the only layer where this gets fixed, and the runtime is something you can build, buy, or demand from your platform vendor today. The alternative is waiting for an agent to read the wrong comment.
