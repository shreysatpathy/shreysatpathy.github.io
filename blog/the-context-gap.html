The Context Gap: Why 67% of Enterprises Can't Trust Their Agents' Answers — And Why the Semantic Layer Is the Most Expensive Infrastructure You Haven't Built

July 1, 2026
AI Infrastructure Data Product Management

Your customer support agent reports that Q2 revenue was $4.2M. Your business intelligence agent, querying the same database, says $3.8M. Your finance agent, looking at the same period, reports $4.5M. Every agent is correct. Every agent gives a different number. Nobody can explain why.

This is not a model hallucination. Every answer traces back to real data. The support agent counted recognized revenue. The BI agent counted booked deals. The finance agent counted cash received. "Revenue" was never defined at the agent layer — each tool inherited whatever schema its connected data source happened to use. The agents returned precise answers to questions their human operators didn't know they were asking differently.

I'm seeing this pattern across every enterprise I talk to. Organizations are deploying multiple agents against the same underlying data stores, and discovering that the same question produces different answers depending on which agent, which tool, or which retrieval path the question traverses. Revenue means one thing in a SQL table, something slightly different in a BI semantic layer, and something else again in an agent instruction file. Nobody built the translation layer between them.

VentureBeat's Q1 2026 Pulse Research quantified what I've been hearing anecdotally: hybrid retrieval architectures — where agents pull from vector stores, SQL databases, BI tools, and knowledge bases simultaneously — tripled from 10.3% to 33.3% of enterprise deployments between January and March. The fastest-growing strategic position in their dataset is also the one most likely to produce contradictory answers. Snowflake addressed this directly at Snowflake Summit 26 with Horizon Context and Cortex Sense — a two-layer system designed to give agents a governed, shared definition of business logic across retrieval stacks. The fact that Snowflake is building this tells you everything about how big the problem has become.

I call this the context gap. Enterprises have data infrastructure. They have agent infrastructure. They don't have the semantic layer that connects them. And the gap is costing them far more than anyone is accounting for.

The three failure modes of context

The context gap manifests in three distinct failure patterns. Every organization I've worked with experiences at least two.

1. Semantic drift across retrieval paths. The same business concept — "customer," "revenue," "active user" — is defined differently in every system an agent touches. Your CRM defines a customer as anyone with a signed contract. Your billing system defines a customer as anyone with a processed payment. Your product analytics defines a customer as anyone with a user account. When an agent queries all three to answer "how many customers do we have?" it gets three different numbers and has no mechanism to reconcile them. The agent doesn't know it's asking different questions. It believes it asked one question and got contradictory answers.

2. Context fragmentation across agent handoffs. As I covered in the coordination gap, multi-agent systems lose context at handoff boundaries. But there's a more insidious version of this: even when agents pass context correctly, they interpret that context through different semantic lenses. Agent A passes "approved deal worth $50K" to Agent B. Agent B's data source defines "deal" as booked-but-unpaid, while Agent A's source defined it as signed-and-validated. The handoff succeeded technically. The meaning was lost in transit.

3. Confident contradictions at scale. This is the most operationally dangerous failure. Each agent returns a precise, well-formatted answer. Each answer is internally consistent. Each answer is confidently wrong relative to what the stakeholder actually asked. The agents are not hallucinating. They are correctly answering questions their human operators didn't ask. The CEO sees a revenue number. The CFO sees a different one. Both came from "the AI." Trust erodes. The pilots that survived production get killed at the next review because leadership can't rely on the outputs.

A recent Prosigns survey of 412 enterprise AI leaders found that 67% of organizations with multi-agent deployments report inconsistent answers to identical questions across different agent tools. Only 12% have any mechanism to detect or reconcile these contradictions. The other 55% are discovering them through executive escalations and board reviews.

Why this is harder than it looks

The obvious fix is "standardize your definitions." Every data team I talk to says this. Every data team has tried. Almost none have succeeded at the agent layer. Here's why.

The semantic layer problem is a coordination problem, not a metadata problem. Most enterprises have data catalogs, business glossaries, and lineage systems. These work for human analysts who can interpret context and ask clarifying questions. They don't work for agents that execute instructions literally. A human reading "revenue" in a SQL column name knows to check the table description. An agent does not — unless the semantic context is embedded in the retrieval path itself. The metadata exists. The runtime integration doesn't.

Hybrid retrieval multiplies the ambiguity. When an agent queries a vector store, a SQL database, and a knowledge base in the same workflow, it isn't just retrieving from different sources — it's retrieving from sources with different ontologies. The vector store indexed documents where "churn" means contract non-renewal. The SQL table defines "churn" as account deletion. The knowledge base article uses "churn" to mean payment failure. The agent synthesizes all three and produces an answer that blends incompatible definitions. The result looks comprehensive. It is comprehensively wrong.

No single vendor owns the fix. This is why the problem persists. Snowflake's Horizon Context solves it for data living in Snowflake. Databricks' Unity Catalog solves it for data in Databricks. Semantic layer vendors solve it for their specific integrations. But enterprise agent workflows cross data platforms as a matter of course. An agent reads customer data from Snowflake, product data from Postgres, and pricing data from a knowledge base — all in the same workflow. No catalog spans all three with runtime enforcement. The market is full of point solutions for a problem that demands a cross-platform standard.

What the winners do

A small cohort of organizations is closing the context gap. The patterns are consistent enough to describe as a framework.

1. They define a runtime semantic layer before they deploy agents. The teams that ship don't start with agent development. They start with a shared ontology — a machine-readable definition of every business concept an agent might encounter, accessible at query time, enforced at the retrieval layer. This isn't a data catalog document that lives in Confluence. It's a live service that every agent calls before it interprets a result. When an agent reads "revenue," the semantic layer returns the definition, the source, and the transformation rules for the specific data path it's querying.

2. They build agent-specific schemas, not just data schemas. Most enterprises define schemas for their databases. Few define schemas for what agents should expect when they query those databases. The winning teams create "agent schemas" — typed interfaces that specify not just the data format but the semantic context, the allowed interpretations, and the reconciliation rules when multiple sources disagree. This is the agent equivalent of an API contract. It turns agent-data interaction from a guessing game into a governed transaction.

3. They test for contradiction, not just accuracy. Standard evaluation frameworks test whether an agent's answer is correct against a ground truth. The winners add a second test: when multiple agents answer the same question, do their answers agree? If not, is the disagreement explainable through legitimate differences in scope, or is it evidence of a context gap? This catch-contradictions-early pattern is the single highest-leverage investment I've seen. The evaluation gap was about whether agents work. The context gap is about whether agents agree. Both need measurement.

4. They invest in semantic drift monitoring. Context gaps don't appear all at once. They emerge as new data sources join the retrieval pool, as agent schemas evolve independently, as business definitions change without updating agent configurations. The teams that sustain multi-agent deployments monitor for semantic drift continuously — tracking whether the same question returns the same answer across agents over time, alerting when divergence exceeds threshold, and treating semantic reconciliation as an ongoing operational practice rather than a one-time data modeling exercise.

The unbudgeted line item

The context gap is not a data problem. It is a product management problem with data consequences. Every PM I know has budgeted for model API costs, infrastructure provisioning, and evaluation tooling. Almost nobody has budgeted for the staff time spent reconciling contradictory agent outputs, the executive trust eroded by inconsistent answers, or the pilot kills that happen because "the AI gives different numbers every time I ask."

The cost gap showed that 98% of enterprises track AI spend but only 9% know what it bought. The context gap explains why: you can track every token, trace every query, and log every output — but if you don't know what "revenue" meant to the agent that returned that number, you don't know what you paid for.

The context layer is the most expensive infrastructure you haven't built. Not because the technology is hard — semantic layers are well-understood — but because fixing it requires coordination across data engineering, product management, and agent development teams that have spent the last 18 months moving in different directions. The organizations that close the context gap will be the ones that treat semantic consistency as a product requirement, not a data infrastructure task. They'll define what "revenue" means before they deploy an agent that reports it. They'll build the ontology before the agent. They'll invest in agreement before accuracy.

Everything else is just confident wrong answers that nobody will catch until the board asks why the AI can't add.
